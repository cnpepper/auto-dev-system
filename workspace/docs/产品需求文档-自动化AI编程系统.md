## 二、需求详情

### [REQ-001] 项目流程编排引擎

**类型：** 功能需求  
**优先级：** P0  
**开发复杂度：** 高

**用户故事：**
> 作为项目经理，  
> 我希望系统能够按照"开发→测试"的流程自动推进项目，  
> 以便减少人工协调成本，实现AI自主开发。

**验收标准：**
- [ ] 系统支持流程定义：开发环节 → 测试环节 → 循环迭代
- [ ] 当一个环节完成后，系统自动触发下一个环节
- [ ] 当测试环节发现问题，系统能够生成需求形式的问题反馈并重新进入技术设计环节
- [ ] 用户可以通过Web界面查看当前所处的环节和状态
- [ ] 支持手动触发流程启动（点击"开始"按钮）
- [ ] 每个步骤记录开始时间和结束时间
- [ ] 系统能够记录执行日志和错误信息

**背景 & 原始反馈摘录：**
> "开发+测试自主完成 → 人类验收 → 反馈bug/改进建议（需求形式）→ 技术设计 → 开发 → 测试 → 循环"
> 
> "MVP核心能力：方案输入 → 开发环节 → 测试环节"
> 
> "流程触发方式：用户手动点击'开始'按钮触发"

**开发备注：**
- 核心目标是确保需求输出→技术设计→开发→测试流程顺畅运转
- MVP阶段只包含开发环节和测试环节，不包含产品设计和技术设计环节

### [REQ-002] 外部AI编程工具适配器

**类型：** 功能需求  
**优先级：** P0  
**开发复杂度：** 高

**用户故事：**
> 作为项目经理，  
> 我希望能够选择不同的AI编程工具（如codebuddy-cli、claude code等）来执行开发任务，  
> 以便根据项目需求灵活选择合适的工具。

**验收标准：**
- [ ] 系统支持至少两种外部AI编程工具的集成（codebuddy-cli、claude code等）
- [ ] 采用适配器模式设计，统一接口抽象
- [ ] 用户可以在系统配置中手动指定使用的工具
- [ ] 工具配置为系统级别配置，全局生效
- [ ] 能够通过接口形式调用外部工具的SDK
- [ ] 工具执行状态能够被系统监控和记录

**背景 & 原始反馈摘录：**
> "系统本身不实现编程功能，通过调用外部编程工具执行开发工作"
> 
> "需要适配多种工具：codebuddy-cli、claude code等"
> 
> "工具选择：用户通过配置手动指定"
> 
> "工具适配：采用适配器模式，统一接口抽象"

**开发备注：**
- 外部工具有专门的SDK，通过接口形式调用
- SDK文档在技术实现阶段提供
- AI速度、上下文等由工具自身配置管理，系统不干预

### [REQ-003] 项目进度状态管理

**类型：** 功能需求  
**优先级：** P0  
**开发复杂度：** 中

**用户故事：**
> 作为项目经理，  
> 我希望系统能够记录和展示当前项目的进度状态、执行日志和历史信息，  
> 以便随时了解项目进展和定位问题。

**验收标准：**
- [ ] 系统能够持久化项目进度状态、工具配置、系统配置
- [ ] 当前进度状态包含：当前环节、当前功能模块、执行状态
- [ ] 记录每个步骤的开始时间、结束时间
- [ ] 记录执行日志、错误信息
- [ ] 一次只运行一个项目（不支持并发多项目）
- [ ] 不需要历史项目列表（只保留当前项目）
- [ ] 用户可以通过Web界面查看完整的进度状态

**背景 & 原始反馈摘录：**
> "需要持久化：项目进度状态、工具配置、系统配置"
> 
> "当前进度状态包含：当前环节、当前功能模块、执行状态"
> 
> "一次只运行一个项目"

**开发备注：**
- 核心目标是需求输出→技术设计→开发→测试流程顺畅运转
- 状态管理是流程编排的基础设施

**背景 & 原始反馈摘录：**
> "方案输入方式：文档（PRD、原型、技术设计）放入指定目录，AI自动获取"

> "MVP不包含：产品设计、技术设计、文档整理环节"

- [ ] 主要任务：按照原型和技术设计实现具体功能
- [ ] 功能可运行、符合原型和接口设计、无报错
- [ ] 需通过单元测试和接口测试
- [ ] AI规范/技能文档存放在项目的docs目录
- [ ] 所有文档（产品文档、技术文档等）统一存放在docs目录

**背景 & 原始反馈摘录：**
> "按功能模块开发，每完成一个功能提交一次代码"
> 
> "主要任务：按照原型和技术设计实现具体功能"
> 
> "验收标准：功能可运行、符合原型和接口设计、无报错"

**开发备注：**
- 不需要人工代码审查
- 依赖外部AI编程工具的执行能力

**类型：** 功能需求  
**优先级：** P0  
**开发复杂度：** 高

**用户故事：**
> 作为项目经理，

- [ ] 使用pytest框架进行E2E测试

- [ ] 测试用例覆盖正向流程、反向流程、异常流程
- [ ] 测试报告生成在docs目录，每轮单独一个报告
- [ ] 人类能够审核测试结果、参与测试、反馈问题
- [ ] 测试失败时，记录问题并反馈给人类

**背景 & 原始反馈摘录：**
> "主要做E2E测试，使用pytest框架"
> 
> "AI根据PRD、技术设计、产品原型开发测试用例"
> 
> "测试报告生成在docs目录，每轮单独一个报告"

**开发备注：**
- 测试发现问题不要自动修复，由人类决策后续处理
- 人类全程参与测试审核

### [REQ-013] 产品设计辅助

**类型：** 功能需求  
**优先级：** P2  
**开发复杂度：** 高

**用户故事：**
> 作为项目经理，  
> 我希望AI能够根据调研结果自动生成PRD和原型，并通过多轮交互逐步细化，  
> 以便快速完成产品设计环节。

**验收标准：**
- [ ] 调研结果 → AI生成PRD → 人类确认/反馈 → 迭代修改 → 确认
- [ ] AI生成原型 → 确认/反馈 → 迭代修改 → 确认
- [ ] 支持多轮交互，逐步细化
- [ ] PRD功能点不遗漏，覆盖正向流程、反向流程、异常流程
- [ ] PRD包含角色、场景、流程、异常处理
- [ ] 原型覆盖所有功能页面/组件，包含各种状态（初始态、加载态、空态、异常态、成功态）

**背景 & 原始反馈摘录：**
> "调研结果 → AI生成PRD → 人类确认/反馈 → 迭代修改 → 确认 → AI生成原型 → 确认/反馈 → 迭代修改 → 确认"
> 
> "MVP不包含：产品设计、技术设计、文档整理环节"
> 
> "第二期及以后：增加产品设计、技术设计、文档整理环节"

**开发备注：**
- 此需求在第二期实现，MVP阶段不包含
- 需要详细的PRD验收标准和原型验收标准

### [REQ-014] 技术设计辅助

**类型：** 功能需求  
**优先级：** P2  
**开发复杂度：** 高

**用户故事：**
> 作为项目经理，  
> 我希望AI能够根据PRD自动生成数据库设计和接口设计，  
> 以便快速完成技术设计环节。

**验收标准：**
- [ ] 主要输出：数据库设计（表结构、字段、索引）、接口设计（API文档、请求/响应格式）
- [ ] 按功能模块呈现
- [ ] 数据库设计输出建表语句
- [ ] 数据库设计不添加外键约束和索引（后期优化），只含主键和必要字段
- [ ] 数据库设计包含通用必要字段：id、status、create_time、update_time等
- [ ] 所有字段需有描述和默认值（主键除外）
- [ ] 接口设计遵循RESTful规范
- [ ] 接口响应格式：错误码code字段 + 统一data字段承载响应数据
- [ ] 接口文档格式：OpenAPI/Swagger（YAML格式）

**背景 & 原始反馈摘录：**
> "主要输出：数据库设计（表结构、字段、索引）、接口设计（API文档、请求/响应格式）"
> 
> "MVP不包含：产品设计、技术设计、文档整理环节"
> 
> "技术栈固定，已明确使用FastAPI全栈模板"

**开发备注：**
- 此需求在第二期实现，MVP阶段不包含
- 技术栈固定为FastAPI全栈模板，不需要技术方案选型

### [REQ-015] 文档整理归档

## 三、非功能需求

### 3.3 兼容性要求
