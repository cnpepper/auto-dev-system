# 数据库迁移使用指南

本文档详细说明了如何使用 SQLModel + Alembic 安全、规范地管理数据库迁移。

---

## 📋 概述

本项目使用 **Alembic** 作为数据库迁移工具，用于管理数据库架构的版本控制和变更。

### 核心工作流程

```
1. 修改 SQLModel 模型代码
   ↓
2. 运行命令：uv run alembic revision --autogenerate
   【自动检测模型变化，自动生成迁移文件】
   ↓
3. 审查自动生成的迁移文件
   ↓
4. 运行命令：uv run alembic upgrade head
   【执行迁移，应用到数据库】
```

### ⚠️ 重要说明

**迁移文件必须使用 `--autogenerate` 自动生成！**

- ✅ **自动生成**：`uv run alembic revision --autogenerate`（必须）
- ❌ **手动编写**：仅在数据迁移、复杂逻辑等特殊情况下使用（错误）

**为什么要自动生成？**
- 避免人为错误导致的数据库结构不一致
- 确保迁移文件与模型定义一致
- 提高开发效率

---

## 🚨 重要警告

### 数据丢失风险

**在执行任何迁移操作前，请务必了解以下风险：**

1. **危险操作会删除数据**
   - `op.drop_table()` - 删除整个表及其所有数据
   - `op.drop_column()` - 删除列及其所有数据
   - `op.execute("DELETE FROM ...")` - 直接删除数据
   - `op.execute("DROP TABLE ...")` - 删除表

2. **本项目的危险迁移案例**
   - `d98dd8ec85a3_edit_replace_id_integers_in_all_models_.py` - 此迁移会删除并重建主键列，如果执行不当会导致数据丢失

3. **迁移失败的风险**
   - 如果迁移执行到一半失败，可能导致数据库处于不一致状态
   - 某些操作（如删除列）是不可逆的，即使有 downgrade 脚本也无法完全恢复

## 📚 Alembic 基础概念

### 迁移文件结构

每个迁移文件包含三个主要部分：

```python
revision = 'abc123'           # 当前迁移的版本号
down_revision = 'def456'      # 前一个迁移的版本号
branch_labels = None          # 分支标签（通常为 None）
depends_on = None              # 依赖的其他迁移

def upgrade():
    """应用迁移的代码"""
    pass

def downgrade():
    """回滚迁移的代码"""
    pass
```

### 迁移链

```
e2412789c190 (初始迁移)
    ↓
d98dd8ec85a3 (ID 从整数改为 UUID) ⚠️ 危险
    ↓
1a31ce608336 (添加级联删除)
    ↓
03cbf8d60028 (迁移到新结构)
```

## 🛠️ 核心操作步骤

### 步骤 1：定义或修改 SQLModel 模型

**文件位置**: `backend/app/models/{module}/{xxx}_model.py`

**命名规范**:
- 表名：小写 + 下划线 + 复数（如 `balance_sheets`）
- 字段名：小写 + 下划线（如 `stock_code`）
- 外键字段：以 `_id` 结尾（如 `owner_id`）
- 时间字段：以 `_at` 结尾（如 `created_at`）
- 布尔字段：以 `is_` 开头（如 `is_active`）

**核心代码示例**:

```python
from datetime import datetime
from typing import Optional
from uuid import UUID, uuid4

from sqlmodel import Field, SQLModel

from ..common.base import BaseModel


# 基础模型（共享字段）
class MyTableBase(SQLModel):
    field1: str = Field(max_length=255)
    field2: Optional[float] = Field(default=None)


# 数据库表模型
class MyTable(MyTableBase, BaseModel, table=True):
    __tablename__ = "my_tables"  # 表名：小写 + 下划线 + 复数
    id: UUID = Field(default_factory=uuid4, primary_key=True)
    created_at: datetime = Field(default_factory=datetime.utcnow)
    updated_at: datetime = Field(default_factory=datetime.utcnow)


# API 响应模型
class MyTablePublic(MyTableBase):
    id: UUID
    created_at: datetime
    updated_at: datetime


# 列表模型
class MyTablesPublic(SQLModel):
    data: list[MyTablePublic]
    count: int
```

**关键注解**:
- `table=True`: 标记为数据库表
- `primary_key=True`: 标记为主键
- `index=True`: 创建索引
- `foreign_key="xxx"`: 外键关联
- `unique=True`: 唯一约束

### 步骤 3：审查迁移脚本

**为什么要审查？**
- Alembic 可能猜测不准确
- 需要确认字段类型、索引、约束
- 检查外键级联删除规则

**审查要点**:
- 表定义检查（字段类型、默认值）
- 索引检查（频繁查询的字段）
- 唯一约束（业务逻辑要求）
- 外键约束（级联删除规则）
- 是否有危险操作（drop_table、drop_column 等）

## 📚 手动编写迁移脚本（特殊情况）

**⚠️ 注意：仅适用于数据迁移、复杂逻辑等特殊情况！**

### 何时需要手动编写迁移？

- 需要数据迁移（数据转换、数据清洗）
- 需要执行复杂的 SQL 逻辑
- `--autogenerate` 无法正确生成的迁移

### 手动编写迁移示例

```bash
# 1. 创建空迁移文件
cd backend
uv run alembic revision -m "manual_migration"

# 2. 编辑迁移文件
```

```python
def upgrade() -> None:
    # 创建表
    op.create_table('my_table', ...)

    # 数据迁移（使用原生 SQL）
    op.execute("""
        INSERT INTO my_table (name, value)
        SELECT name, value FROM old_table
        WHERE status = 'active'
    """)

def downgrade() -> None:
    # 回滚数据迁移
    op.execute("DELETE FROM my_table")

    # 删除表
    op.drop_table('my_table')
```

### 数据迁移模式

```python
def upgrade():
    """
    安全的数据迁移模式：
    1. 添加新列（nullable=True）
    2. 迁移数据
    3. 设置新列为 not null
    4. 删除旧列
    """
    # 步骤 1: 添加新列
    op.add_column('user', sa.Column('new_email', sa.String(), nullable=True))

    # 步骤 2: 迁移数据
    connection = op.get_bind()
    connection.execute("UPDATE user SET new_email = email WHERE email IS NOT NULL")

    # 步骤 3: 设置为 not null（先确保没有 NULL 值）
    connection.execute("UPDATE user SET new_email = '' WHERE new_email IS NULL")
    op.alter_column('user', 'new_email', nullable=False)

    # 步骤 4: 删除旧列
    op.drop_column('user', 'email')
    op.alter_column('user', 'new_email', new_column_name='email')
```

### 1. 迁移前检查清单

- [ ] 验证备份文件完整性
- [ ] 确认备份可以成功恢复

#### ✅ 代码审查
- [ ] 迁移脚本经过代码审查
- [ ] 在测试环境验证过
- [ ] 有完整的 downgrade 脚本

#### ✅ 影响评估
- [ ] 评估迁移对现有数据的影响
- [ ] 评估迁移对应用的影响
- [ ] 制定回滚计划

### 2. 迁移最佳实践

#### ✅ DO（推荐做法）

1. **使用自动生成迁移**
   ```bash
   # 自动生成迁移脚本（推荐）
   uv run alembic revision --autogenerate -m "描述迁移内容"
   ```

2. **手动编写安全的迁移（特殊情况）**
   ```python
   def upgrade():
       # 添加新列（安全）
       op.add_column('user', sa.Column('new_field', sa.String(), nullable=True))

       # 创建新表（安全）
       op.create_table('new_table', ...)

       # 添加索引（安全）
       op.create_index('ix_user_email', 'user', ['email'])
   ```

3. **数据迁移使用事务**
   ```python
   def upgrade():
       connection = op.get_bind()
       connection.execute(
           "UPDATE user SET status = 'active' WHERE status IS NULL"
       )
   ```

4. **使用批量操作处理大量数据**
   ```python
   def upgrade():
       connection = op.get_bind()
       # 分批处理，避免锁表时间过长
       for offset in range(0, 100000, 1000):
           connection.execute(
               "UPDATE large_table SET field = 'value' LIMIT 1000 OFFSET %s",
               offset
           )
   ```

#### ❌ DON'T（禁止做法）

1. **不要在迁移中直接删除表或列**
   ```python
   # ❌ 危险：直接删除表
   def upgrade():
       op.drop_table('user')

   # ✅ 安全：先标记为废弃，后续再删除
   def upgrade():
       op.add_column('user', sa.Column('is_deprecated', sa.Boolean(), default=False))
   ```

2. **不要在迁移中修改数据类型而不考虑数据转换**
   ```python
   # ❌ 危险：直接修改列类型，可能导致数据丢失
   def upgrade():
       op.alter_column('user', 'age', existing_type=sa.Integer(), type_=sa.String())

   # ✅ 安全：先添加新列，迁移数据，再删除旧列
   def upgrade():
       op.add_column('user', sa.Column('age_str', sa.String(), nullable=True))
       connection.execute("UPDATE user SET age_str = CAST(age AS VARCHAR)")
       op.drop_column('user', 'age')
       op.alter_column('user', 'age_str', new_column_name='age')
   ```

3. **不要在迁移中执行长时间运行的 SQL**
   ```python
   # ❌ 危险：长时间锁表
   def upgrade():
       connection.execute("UPDATE huge_table SET field = 'value'")

   # ✅ 安全：使用批量处理或后台任务
   ```

## ⚠️ 常见问题

### Q1: Alembic 检测不到模型变更？

**原因**: 模型定义没有使用 `table=True` 或导入路径不正确

**解决方案**: 检查 `alembic/env.py` 中的 import 语句

### Q2: 表已存在错误？

**原因**: 数据库中已有表，但迁移记录缺失

**解决方案**:
```bash
# 方案 1：标记当前状态
uv run alembic stamp head

# 方案 2：手动删除表后重试（会丢失数据）
psql -U postgres -d app -c "DROP TABLE IF EXISTS my_table;"
uv run alembic upgrade head
```

```bash
# 标记迁移为已完成（不执行）
uv run alembic stamp <revision_id>
```
